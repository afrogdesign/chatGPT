## 🏫 この案件の位置づけ

今回の仕組みは、継続利用される音楽サービスではなく、**大学LPに組み込まれる期間限定の体験装置**です。

つまり目的は次のようになります。

- 大学の世界観を感じてもらう
- 訪問者の印象に残る
- SNSで軽く話題になる
- 「触れるLP」として体験価値を上げる

このため、必要なのは高機能な作曲システムではなく、次の4点です。

- すぐ生成されること
- 見た目が気持ちいいこと
- 危ない使われ方をされにくいこと
- 利用条件が明確であること

> [!tip]
> この案件では、音楽の完成度そのものよりも、**LPとしての気持ちよさ**と**大学らしい世界観**のほうが重要です。

## 🏗️ 最適なシステム構成

もっとも軽くて現実的な構成は次の形です。

```text
[LPページ]
  ↓
[単語入力UI]
  ↓
[APIサーバー]
  ├─ 入力チェック
  ├─ 単語→音楽プロンプト変換
  ├─ ジョブ発行
  ↓
[音楽生成API]
  ↓
[一時保存]
  ↓
[LP上で再生]
```

### この構成で十分な理由

- 会員登録が不要
- マイページが不要
- 楽曲ライブラリが不要
- 決済が不要
- 長期保存が不要
- 音楽編集機能が不要

必要な画面は、実質的に次の5つだけです。

1. 入力欄
2. 生成ボタン
3. ローディング演出
4. 再生UI
5. 注意書き

## 🧭 体験設計のポイント

### 自由入力だけにしない

完全自由入力だけにすると、出力が不安定になったり、危険なワードが混じったりしやすくなります。

そのため、**自由入力 + 補助UI** の組み合わせが最適です。

### 入力UIの例

#### 単語入力

```text
静けさ / 夜 / 雨 / 芸術 / 余白
```

#### 補助UI

- 雰囲気：静か / 明るい / 不思議 / 力強い
- 長さ：15秒 / 30秒
- 音色：ピアノ / 電子音 / アンビエント
- 用途：展示 / 散歩 / 読書 / 空想

### なぜこの設計がよいか

| 項目 | 理由 |
|---|---|
| 自由入力 | 体験として面白い |
| 補助タグ | 出力を安定させやすい |
| 長さ選択 | コストと待ち時間を制御しやすい |
| 雰囲気選択 | LPの世界観に寄せやすい |

---

## 生成結果の見せ方

生成後の表示は、単に音声プレイヤーを置くだけでは弱いです。

### おすすめ表示要素

- 自動生成されたタイトル
- 波形風または音に反応するビジュアル
- 再生ボタン
- 「もう一度つくる」ボタン
- 「少し似た感じでもう一回」ボタン
- 個人利用のみの注記

> [!info]
> このタイプのLPは、完成音源よりも**生成されるまでの演出**と**出た瞬間の気持ちよさ**が体験価値になります。

## ⚖️ 利用条件と権利表示

大学LPでのキャンペーン用途なら、利用条件はかなり明示的に書いておくべきです。

### 表示すべき内容

| 項目 | 明記する内容 |
|---|---|
| 利用範囲 | 個人利用の範囲でのみ使用可能 |
| 禁止事項 | 商用利用、再配布、再販売、権利主張 |
| 保存 | 生成データは一定期間後に削除される場合がある |
| 出力特性 | AI生成のため毎回異なる結果になる |
| 類似性 | 既存作品との類似が完全に排除されるわけではない |
| 責任範囲 | 大学側は生成結果の完全性や独占性を保証しない |

### 設計上の考え方

- 大学側のルールだけで完結しない
- 利用する生成API側の規約にも従う必要がある
- LP内では「個人利用のみ」を明確に強く出す
- 音源を資産として渡すのではなく、体験として提供する

> [!warning]
> 「うちのLPでは個人利用のみ」と書いていても、元の生成基盤の規約より強い権利を配れるわけではありません。  
> 必ず、利用する生成API側の条件も前提にして整理する必要があります。


#### generation_assets

```text

duration_sec

```

#### access_logs

```text
id

ip_hash
user_agent
created_at
```

### 今回は不要になりやすいもの

- 課金情報
- 恒久保存ライブラリ
- プレイリスト
- 重い所有権管理
- マイページ機能

> [!note]
> 今回の中心は「作品資産の蓄積」ではなく、**その場で完結する体験**です。  
> したがって、保存設計も軽くして問題ありません。

## 🔐 安全設計と制限

LPキャンペーンでも最低限の安全設計は必要です。

### 必須項目

- 文字数制限
- 禁止語フィルタ
- 有名アーティスト名の検知
- 連続生成回数制限
- bot対策
- 同時アクセス制限
- タイムアウト制御

### 特に重要な3点

#### 1. 有名アーティスト名を避ける

「〇〇風」「〇〇みたいに」といった入力は事故の元になりやすいです。

#### 2. 長尺を避ける

長い音源ほどコストも待機時間も増え、LP体験が重くなります。  
キャンペーンなら15〜30秒で十分です。

#### 3. 保存性を弱くする

場合によってはダウンロード不可にして、**その場で聴くだけ**にする設計も有効です。

> [!tip]
> もっとも安全なのは、**生成してその場で体験できるが、資産としては持ち帰りにくい**設計です。

## 🚀 もっとも現実的なMVP

### ユーザーができること

- 単語を3〜5個入れる
- 雰囲気を選ぶ
- 15秒 or 30秒を選ぶ
- 生成する
- その場で聴く
- もう一度生成する

### 管理側が持つもの

- 管理画面でON/OFF
- 1日あたり生成上限
- アクセス数確認
- 不正入力ログ
- 利用条件文面の差し替え

### 画面構成

| 画面 | 内容 |
|---|---|
| ヒーロー | キャッチコピー、開始ボタン |
| 生成フォーム | 単語入力、雰囲気選択、長さ選択 |
| ローディング | 単語が音へ変換される演出 |
| 結果画面 | 再生、再生成、シェア導線、注記 |
| フッター | 利用条件、禁止事項、削除方針 |

### 実装方針

| 項目 | 推奨 |
|---|---|
| フロント | Next.js |
| バックエンド | Node.js / TypeScript |
| 非同期処理 | Queueあり |
| 保存先 | S3互換ストレージ |
| セッション管理 | cookieベース |
| 認証 | 原則なし |
| CDN | あり |
| アクセス対策 | rate limit + captcha |

## 📌 まとめ

| 観点 | この案件での最適解 |
|---|---|
| プロダクトの正体 | LP内の体験型コンテンツ |
| 音楽の役割 | 世界観を体験させる短いアート/BGM |
| 技術構成 | 外部生成API + 軽量APIサーバー |
| 保存戦略 | 一時保存のみ |
| ユーザー管理 | なし、または最小限 |
| 安全性 | アーティスト名禁止、回数制限、注意書き強化 |
| 権利整理 | 個人利用のみを明記し、元API規約も前提にする |
| 最重要ポイント | 音楽機能の多さではなく、LP体験の完成度 |

---

> [!abstract]
> この案件では、音楽生成機能を本格的なサービスに育てる必要はありません。  
> むしろ、**大学の世界観を来訪者の言葉から短い音に変える装置**として設計するほうが、目的にも予算にも運用にも合っています。  
> したがって、設計の核心は **高機能化ではなく、体験の質・安全性・権利表示の明快さ** にあります.
